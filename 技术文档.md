# 一、项目定义

## 项目名称

**面向学术论文 PDF 的多阶段检索与结构化解读 Agent**

## 一句话定义

构建一个面向学术论文 PDF 的 Agent：它能先判断用户是在"找论文、读论文，还是追问章节"，再动态调用论文级和章节级检索工具，完成从"找论文"到"读论文"再到"章节追问"的完整闭环。

## 项目目标

解决传统论文问答系统里的几个典型问题：

- 用户问题意图多样，不适合单一路径检索
- 学术 PDF 结构复杂，标题、摘要、方法、实验、公式、表格分布在不同层级
- 直接对整篇论文做问答，容易出现长上下文利用不稳定的问题
- 用户往往不是只问一轮，而是会基于当前论文继续追问

因此，系统不做"固定 top-k 检索 + 一次性生成"，而是做成一个**多阶段、可控、带状态的论文解读 Agent**。

------

# 二、为什么定义成 Agent，而不是普通 RAG

这个项目完全可以合理地定义成 Agent，因为它具备 Agent 的几个关键特征：

- **意图识别**：先判断用户到底在问什么类型的问题
- **工具选择**：按任务调用不同检索工具，而不是固定检索
- **状态管理**：记录当前锁定论文、当前关注章节
- **多阶段决策**：论文未锁定时先找候选，锁定后再深入正文
- **按需扩展上下文**：不是默认读全文，而是先小范围读，再必要时扩展
- **多轮延续**：支持"接着上一轮继续问"

所以项目的核心不是"模型自己无限思考"，而是：

**你设计了一个具备任务路由、工具调用、上下文控制和状态更新能力的轻量 Agent。**

------

# 三、核心用户场景（V1）

V1 聚焦 3 个核心场景，构成"找论文 → 读论文 → 章节追问"完整闭环。

## 1. 模糊找论文

### 用户示例

- 那篇会反思检索结果的 self-rag 论文是什么？
- 有篇论文是用图做检索增强的，叫什么？
- 我记得有篇讲 query routing 的 RAG 论文

### Agent 行为

- 判断这是**模糊找论文**
- 只检索论文级字段：
  - 标题
  - 摘要
  - 关键词
  - 作者
  - 年份
- 返回候选论文卡片
- 若候选不唯一，等待用户确认或自动根据上下文继续缩小范围

### 输出形式

- 论文标题
- 作者
- 年份
- 一句话摘要
- 匹配理由

------

## 2. 指定论文解读

### 用户示例

- 给我讲讲 Self-RAG 这篇论文
- 解释一下这篇论文的核心思路
- 这篇论文解决了什么问题？

### Agent 行为

- 识别到用户已经锁定论文
- 切换到**章节级阅读**
- 优先读取：
  - title
  - abstract
  - introduction
  - method
  - experiments
  - conclusion
- 输出结构化解读

### 输出形式

- 研究问题
- 核心贡献
- 方法框架
- 实验结果
- 局限性

------

## 3. 章节定向解读（追问）

### 用户示例

- 只讲这篇论文的方法部分
- 它的创新点是什么？
- 这篇论文的实验设置是什么？
- 它和 prior work 最不一样的地方是什么？

### Agent 行为

- 判断为**章节级定向解读**
- 在锁定论文后，按 section type 检索
- 只读与用户问题相关的 section
- 输出定向回答

### 输出形式

- 方法流程
- 创新点
- 实验配置
- 消融结论
- 相比 baseline 的差异

------

# 五、论文 PDF 解析方案

这是这个项目区别于普通 RAG 的关键之一。

## 5.1 解析目标

不是单纯把 PDF 转成纯文本，而是尽可能提取：

- paper metadata
- 章节结构
- 正文段落
- 表格
- 参考文献

## 5.2 V1 解析栈

### GROBID（V1 唯一解析器）

GROBID 专门面向技术和科学出版物，目标是把 PDF 解析成结构化的 TEI/XML；官方文档明确说明它聚焦 scientific publications，fulltext 模型会识别段落、section titles、figures、tables、formulas 等对象。

V1 只用 GROBID，负责：
- metadata 提取（标题、作者、摘要、关键词、年份）
- 章节结构恢复（section titles + paragraphs）
- 表格基础提取
- 参考文献提取

## 5.3 为什么不把 OCR 作为重心

因为这个项目的核心对象是**born-digital 学术论文 PDF**，所以重点不再是"脏扫描件 OCR 清洗"，而是：

- 章节恢复
- 表格对象化
- caption 与正文对齐
- 论文级到章节级的层次索引

------

# 六、为什么不能做"整篇论文一次性灌给模型"

这点要写进技术方案，因为它决定你的核心设计。

长上下文模型的标称窗口已经很大，Gemini 官方文档也提供了 1M tokens 级上下文的说明；但"窗口大"不等于"对整篇长文的任意位置都能稳定利用"。Lost in the Middle 这类研究表明，当关键信息落在输入中间位置时，模型性能会明显下降。

所以本项目的"全文解读"不是：

**整篇论文一次性喂给模型 + 任意问答**

而是：

**受控全文解读（controlled whole-paper reading）**

也就是：

1. 先读最关键的全局部分
   - title
   - abstract
   - conclusion
2. 再按用户问题补读目标 section
   - method
   - experiments
   - related work
   - ablation

这样既更稳，也更符合长上下文使用的最佳实践。Google 官方 long-context 文档本身也强调，开发者需要主动优化 long context 的使用方式。

------

# 七、上下文优化机制

这部分是你的方案亮点，建议重点写。

## 7.1 分层阅读

### 第一层：论文级阅读

读取：

- 标题
- 摘要
- 关键词
- 结论
- 章节目录

适用：

- 模糊找论文
- 初始解读

### 第二层：章节级阅读

按问题只读相关 section：

- introduction
- method
- experiment
- conclusion

适用：

- 指定论文解读
- 章节定向解读

## 7.2 先证据后回答

长文档任务里，先抽证据、再生成答案会更稳。Anthropic 的官方提示工程文档建议对复杂上下文使用清晰结构和标签；把文档和元数据明确包起来，能减少模型混淆。

所以生成协议建议固定成：

1. 先抽取 evidence snippets
2. 再整理 explanation
3. 最后输出 citations

## 7.3 文档结构标签化

输入模型时，不要塞一坨文本，建议用结构化标签，例如：

```
<paper>
  <metadata>
    <title>...</title>
    <authors>...</authors>
    <year>...</year>
  </metadata>
  <section type="method" page_start="5" page_end="8">
    ...
  </section>
</paper>
```

这种层次化标签设计与 Anthropic 对复杂 prompt 的 XML 标签建议是一致的。

## 7.4 会话状态管理

至少保存这些状态：

- `candidate_papers`
- `current_paper_id`
- `current_focus_section`
- `last_intent`

这样多轮追问才能成立。

------

# 八、Agent 状态机设计

建议把 Agent 做成**轻量状态机**，而不是自由规划型重 Agent。

## 8.1 状态定义

### `S0_ROUTING`

识别用户意图，分类为：找论文 / 读论文 / 追问章节。

### `S1_SEARCHING`

论文级检索，返回候选论文卡片。

### `S2_LOCKED`

已锁定目标论文，准备进入深读。

### `S3_READING`

章节级深读，包含指定论文解读和章节定向解读。

### `S4_ANSWERING`

结构化回答生成 + 更新会话状态（生成和状态更新作为该状态的终点动作，不单独拆状态）。

## 8.2 状态转移逻辑

### 模糊找论文

```
ROUTING -> SEARCHING -> (候选确认) -> LOCKED
```

### 指定论文解读

```
ROUTING -> LOCKED -> READING -> ANSWERING
```

### 章节追问

```
ROUTING -> LOCKED -> READING -> ANSWERING
```

### 状态回退

支持在任意状态回退：
- 用户在 READING 状态问了一篇新论文 → 回退到 SEARCHING
- 用户在 LOCKED 状态切换论文 → 回退到 SEARCHING
- 用户在 ANSWERING 后追问同一论文的其他章节 → 回退到 READING

# 十二、受控全文解读机制

"全文解读"要做，但要做成受控式。

## 12.1 V1 的全文解读定义

全文解读 =

- 先读 `title + abstract + conclusion`
- 再按需补读 `method / experiments`

而不是"一次性整篇读完"。

## 12.2 全文综合摘要流程

### Step 1：全局摘要

从：

- title
- abstract
- conclusion
  生成全局 summary

### Step 2：章节摘要

对：

- method
- experiments
  分别生成 section summary

### Step 3：统一综合

把上述 summary 合成为：

- 核心问题
- 方法
- 实验
- 优势
- 局限

## 12.3 为什么这样做

这能显著降低：

- 中间信息丢失
- prompt 过长
- 关键证据被淹没
- 任意问答时上下文不稳

这和长上下文研究对"输入位置敏感性"的发现是吻合的。

# 十九、评测设计

V1 聚焦两个核心评测维度 + 一个对比实验。

## 19.1 意图路由准确率

- 50 条评测集的 intent routing 分类准确率
- 按场景分别统计

## 19.2 端到端回答质量（LLM-as-a-Judge）

使用独立 LLM（如 GPT-4）对生成回答打分，不用系统本身的生成模型评自己。

评分维度（1-5 分）：
- **Faithfulness**：回答是否忠于原文
- **Relevance**：回答是否切题

评分流程：
1. LLM 按预设 rubric 对每条回答打分
2. 人工对 10-20 条样本同时打分
3. 计算 LLM 评分与人工评分的一致性（Cohen's Kappa 或 Spearman 相关系数）
4. 一致性达标后，用 LLM 评分覆盖全量评测集

## 19.3 A/B 对比实验

构建 flat RAG baseline（普通 chunk + top-k 检索 + 直接生成），用同一批评测问题对比：
- 分层 Agent vs. flat RAG baseline
- 对比指标：Faithfulness 和 Relevance 平均分
- 输出效果差异表，作为面试核心数据支撑

# 二十二、项目亮点

这版方案真正的亮点不是"我也做了一个论文 RAG"，而是：

1. **把系统定义成论文解读 Agent，而不是单轮 QA**
2. **检索是分层的：论文级、章节级，不是 flat chunking**
3. **全文解读是受控的，不做 naive full-context**
4. **轻量状态机支持多轮追问和状态回退**
5. **支持从找论文到读论文再到章节追问的完整闭环**
6. **有 A/B 对比数据：分层 Agent vs. flat RAG baseline**

------

# 二十三、简历版表述（STAR 法则）

## Situation

传统论文问答系统采用 flat chunking + top-k 检索的固定管道，无法区分"找论文""读论文""追问章节"等不同意图，且直接将整篇论文灌入长上下文存在信息利用不稳定的问题。

## Task

设计并实现一个面向学术论文 PDF 的多阶段检索与解读 Agent，支持意图驱动的分层检索和多轮追问。

## Action

- 基于 GROBID 将论文 PDF 解析为结构化数据，构建**论文级 + 章节级两层索引**（dense + BM25 混合召回 + BGE-reranker 排序）
- 设计**5 状态轻量状态机**实现意图路由与工具调用，支持"找论文 → 读论文 → 章节追问"闭环及状态回退
- 采用**受控全文解读**机制（先 abstract/conclusion 建框架，再按需补读 method/experiment），替代 naive full-context
- 实现 **evidence-first 生成协议**，先抽取证据再组织回答，提升忠实度与可追溯性

## Result

- 在 RAG 领域 20+ 篇论文、50 条评测集上，意图路由准确率达 X%
- 相比 flat RAG baseline，LLM-as-a-Judge 评分中 Faithfulness 提升 X%，Relevance 提升 X%
- 支持完整的多轮追问闭环，Demo 展示意图路由、检索命中与 evidence 溯源可视化
