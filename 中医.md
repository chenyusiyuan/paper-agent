可以借鉴，而且我觉得**你这个痛点和那个中医项目的核心矛盾其实是同一个**：

**既不能“抽太少”，导致检索不到细节；也不能“全量直接喂”，导致噪声大、上下文失控。**

那个中医项目真正可借鉴的，不是某个具体 parser 技巧，而是它背后的处理原则：

**解析尽量保留全量信息，检索分层筛选，回答按需深读。**

你当前文档其实已经有这个方向了：V1 是**论文级索引 + 章节级索引**，指定论文解读先读 `abstract + conclusion` 建全局框架，再按需补 `method / experiments`，并明确不做 naive full-context。也就是说，你的主架构本身没有错，问题主要出在 **parser 太“瘦”了**，导致“章节级索引”实际上只拿到了一小部分正文。

## 先把问题说透

你现在担心的其实不是“要不要滚动阅读”，而是：

1. **如果 parser 只取一级 div 甚至只取第一个 div，很多 method/experiment 细节根本不会进索引。**
2. **如果 parser 把全文都粗暴展开，又会把噪声、嵌套结构、caption、references 一起带进来，后面检索和生成会变乱。**

这两个担心都对。
 所以正确答案不是二选一，而是：

> **全量保留，但分层使用。**

这也是我认为那个中医项目大概率真实采用的办法：
 它不会只保留“最前面一小段”，而是会先把文档尽可能完整地恢复成一个**可回溯的中间表示**，然后再通过**语义切分 + token 回退 + 混合检索 + 二次过滤**控制真正进入模型的证据量。换到你的论文项目里，思路是一样的，只不过噪声从 OCR/古籍版面，变成了**学术 PDF 的层级结构、caption、嵌套小节、附录和参考文献**。这是场景相似但不完全相同的地方。

## 你最该借鉴的，不是“滚动阅读”，而是“三层视图”

我建议你把原始 PDF 处理分成三层，不要让 parser 直接决定模型看什么。

### 第一层：归档层（全量保留）

这一层的目标只有一个：

**把 PDF 尽可能完整、结构化地存下来，便于回溯。**

这里你应该做的是：

- 保存原始 GROBID TEI/XML
- 递归解析所有 `div/head/p`，而不是只取一级 div
- 保留 section hierarchy（章节、子章节、子子章节）
- 保留 page range
- 保留 raw section title
- 保留 caption / table / figure anchor
- references / appendix 不删除，但打低优先级标签

这一层**不直接给 LLM 用**，它只是“完整表示”。

这一步就是你当前 parser 最需要升级的地方。你文档里现在的 V1 解析目标已经写了要做 metadata、章节结构、正文段落和表格基础提取，所以往“更完整的结构化表示”走，本身和你当前方案并不冲突。

### 第二层：索引层（分层可检索）

这一层才是给检索系统用的。

这里不要理解成“section index = 每个大章节一整块文本”。
 **章节级索引完全可以做成“同一层索引里有粗粒度和细粒度两种 chunk”。**

这是解决你痛点的关键。

我建议你这样改：

#### paper index

继续保持现在的设计，用来解决“找论文”：

- title
- abstract
- keywords
- authors
- year
- venue

这个你文档里已经定了。

#### section index（但内部双粒度）

把 section index 改成两种 granularity，共用一个索引：

1. **coarse chunk**
   - 一个 major section 一个或少量 chunk
   - 例如整段 method、整段 experiments 的摘要式 chunk
   - 用于全局定位
2. **fine chunk**
   - 在 section 内继续按 subsection / paragraph / token window 切
   - 例如 method 下的一个子方法段、experiment 下的一组实验设置段
   - 用于细致追问

也就是说，你仍然符合“V1 只有两层索引”的文档边界，但 section index 本身要变成**层次化 section index**，而不是“一节 = 一块”。这一步非常重要，因为它能在不推翻你现有架构的前提下，让系统重新具备“细致分析一篇论文”的能力。

### 第三层：阅读层（按需深读）

这一层才是喂给 LLM 的东西。

这里的原则是：

**离线尽量全，在线尽量小。**

也就是：

- 不把整篇论文一次塞进去
- 先给全局框架
- 再根据问题补局部证据
- 证据不够再继续局部展开

这和你文档里写的“受控全文解读”是一致的：先读 `title + abstract + conclusion`，再按问题补 `method / experiments`，只不过我要补一句——**这个“补”应该在选中的 section 内继续往下滚，而不是整篇往下滚。**

## 所以，“滚动阅读”怎么改才对

你现在的“滚动阅读”思路方向没错，但单位应该改。

### 不推荐

整篇论文从头到尾滚动窗口阅读。

问题：

- 太浪费 token
- middle sections 仍然会淹没
- query 不相关的内容很多
- references / appendix 容易进来

### 推荐

**在“已选中的 section”内部滚动阅读。**

也就是：

1. 先论文级定位
    用 paper index 找对论文
2. 再 section 级定位
    用 coarse section chunks 找到最相关的大节（method / experiment / conclusion）
3. 再在该节内部做局部 rolling
    在 fine chunks 上按顺序取：
   - 命中 chunk
   - 前 1 个相邻 chunk
   - 后 1 个相邻 chunk

这样做的好处是：

- 保留局部连续性
- 不会因为切块太碎而失去上下文
- 又不会把整篇内容都带进 prompt

这其实就是把中医项目里的“语义切分 + token 回退 + 动态检索”翻译成你的论文场景版本。

## 我给你的最终解决方案

如果只说一句核心策略，就是：

> **存全量、索引用中层、回答用局部。**

展开成工程方案就是下面这套。

### A. parser 层要升级，但不是升级成“全文直灌器”

你现在应该把 parser 升级成：

- 递归解析所有 nested section
- 保留 `section_path`
- 保留 `level`
- 保留 `parent_section_id`
- 保留 `order_in_paper`
- 保留 `page_start / page_end`
- 保留 raw title 和 normalized section type
- references / appendix / ack 默认不进主检索，但保留原文

建议你把原来的 `parse_tei_to_paper()` 产物拆成两份：

1. `parsed_raw/{paper_id}.json`
   - 完整层级结构
   - 尽量保留全量
2. `normalized/{paper_id}.json`
   - 供索引使用
   - 做好 section type 归一化
   - 做好 chunk 化

### B. section index 不要只做“大章节一块”

这是你现在最关键的一刀。

建议 `SectionChunk` 多一个字段：

- `granularity: coarse | fine`

然后：

- coarse：major section summaries / section-level chunks
- fine：subsection / paragraph / token-window chunks

这样你还能保持现在技术文档里的“只有 paper index 和 section index”框架，不需要额外引入第三层索引。

### C. query-time 采用两阶段深读

不同场景用不同策略。

#### 模糊找论文

只查 paper index。
 这部分不受 parser 粒度影响太大。

#### 指定论文解读

流程建议改成：

1. 先取：
   - title
   - abstract
   - conclusion
   - section map
2. 从 coarse section chunks 里选 1–2 个最相关 section
    通常是 method / experiments
3. 再到对应 section 的 fine chunks 里做局部检索
    取 top-k + 邻接窗口
4. 先证据、后回答

#### 章节定向追问

流程建议改成：

1. 锁定当前 `paper_id`
2. 锁定 `current_focus_section`
3. 只在该 section 的 fine chunks 里检索
4. 加前后相邻窗口
5. 若证据不足，再回退到 coarse chunk 或相邻 section

这就是你项目里真正应该实现的“滚动阅读”。

### D. prompt 只吃“证据包”，不吃解析结果全集

这一点一定要坚持。

每次回答，构建一个 evidence pack：

- global frame
   `title + abstract + conclusion + section titles`
- local frame
   `selected fine chunks + adjacent chunks`

然后用你文档里已经写好的 evidence-first 协议去回答。

## 这和中医项目最像的地方在哪

真正相像的不是“论文和古籍都很长”，而是：

### 共同点

- 原始文档都不该直接喂模型
- 原始文档都必须先变成**可回溯的结构化表示**
- 检索必须分层
- 回答必须只基于被筛出来的局部证据
- 证据不足时必须继续检索，而不是硬答

### 不同点

- 中医项目更难的是 OCR 脏文本、异体字、跨格式和领域术语
- 你的论文项目更难的是 section hierarchy、caption/object 对齐、附录/参考文献污染

所以你可以借鉴它的**处理原则**，但不要照搬它的“多格式 + OCR 优先”那一套。

## 如果你问我现在最该改什么

我会给你一个非常明确的优先级：

### 第一优先级

**把 parser 从“只取一级 div”升级成“递归 section tree + normalized section type”。**

### 第二优先级

**把 section index 改成 coarse/fine 双粒度，但逻辑上仍叫 section index。**

### 第三优先级

**把滚动阅读限制在“选中的 section 内部”，不要对整篇做滚动。**

这三步做完，你的系统会立刻从“能找论文但读不深”，变成“能找论文，也能对单篇论文做相对细致的局部深读”。

## 最后给你一个简化版结论

你这个项目的核心痛点，解决办法不是：

- 只抽一点
   也不是
- 全部都 parse 后全都喂进去

而是：

**先尽量完整地 parse，保存结构；再做分层 chunk；再按 query 先粗定位 section，后细读 section 内部；最终只把局部证据包喂给 LLM。**

这就是我认为你最该借鉴那个中医项目的地方。













这段项目介绍如果要**真正落地**，本质上不是一个功能，而是 **5 条子系统同时工作**：

1. 文档解析与建库
2. 混合检索与重排
3. 检索决策 Agent
4. 分层记忆
5. 自动化评测与追踪

实际工程里，通常会先做 **V1 主链路**：

**文档入库 → 切分 → 建索引 → 检索 → 重排 → 证据判断 → 生成回答 → 引用出处**

然后再叠加：

- 动态检索 Agent
- 记忆系统
- 自动评测闭环

所以这段描述如果是真实项目，最可能的落地方式不是“一次把所有点都做完”，而是 **V1 跑通主链路，V2/V3 再加 Agent 和记忆**。

# 二、第一段：多格式文档解析流水线是怎么实现的

你这段里写的是：

> 构建多格式文档解析流水线（PDF / MD / JSON 等），结合语义切分与 Token 回退策略，使得长文信息保留率提升约 20%。

这在实际里通常会拆成 4 步。

## 1. 文档读取

常见实现：

- **PDF**：PyMuPDF / pdfplumber
- **扫描 PDF**：PaddleOCR / OCRmyPDF
- **MD / JSON**：直接读结构化文本
- **教材 / 古籍**：通常 PDF 为主，少部分 JSON/MD 做补充

这里真正的难点不是“读文件”，而是**把原始版面恢复成适合切分的文本结构**。

例如会做这些清洗：

- 去页眉页脚
- 去页码
- 去重复空行
- 合并被 PDF 切断的段落
- 处理 OCR 错字和断行
- 保留标题层级、页码、章节路径

------

## 2. 结构恢复

为了后面能做“引用式回答”，每个 chunk 往往都会保留 metadata，例如：

- `doc_id`
- `title`
- `page`
- `section_path`
- `source_type`（古籍/教材/医案）
- `is_ocr`
- `ocr_confidence`

这一步很重要，因为后面：

- rerank 要用
- 引用要用
- 错误分析要用

------

## 3. 语义切分

不是按固定 500 字硬切，而是优先按：

- 标题
- 小节
- 自然段
- 列表
- 问答对
- 医案中的病例结构

来切。

这就是你简历里说的“语义切分”。

------

## 4. Token 回退策略

如果某个结构块太长，例如超出 embedding 或 reranker 的理想长度，就继续往下拆：

- 先按小标题拆
- 再按段落拆
- 再按句子拆

这就是“Token 回退”。

### 这 20% 信息保留率提升通常怎么测

不是玄学，而是用一套带 gold evidence 的问题集，对比：

- 固定窗口切块
- 结构切分 + token 回退

看 gold evidence 是否更容易被召回。

所以这个“20%”通常不是回答准确率本身，而更像是：

- evidence coverage
- answer support coverage
- 或 retrieval hit rate 的提升

# 三、第二段：混合召回与重排是怎么实现的

你写的是：

> 设计混合召回与重排检索架构，并引入二次过滤机制提升证据质量。有效抑制 OCR 错误和低价值证据，使 Recall@30 从 62% 提升至 84%，Top-1 准确率提升 13%，Faiss-HNSW 索引 P95 延迟保持在 120ms。

这部分是整个系统最核心、最真实的一段。

## 1. 混合召回

一般会同时做两路：

### 稠密检索

- embedding 模型：BGE 类
- 向量库：Faiss-HNSW

作用：

- 解决语义表达不一致
- 比如“咳嗽痰黄”能召回“痰热壅肺”

### 稀疏检索

- BM25 / Elasticsearch / rank-bm25

作用：

- 解决术语、方剂名、证型名、疾病名、编号类问题
- 对 OCR 轻微错误也有一定容忍度

### 融合方式

通常用：

- RRF（Reciprocal Rank Fusion）
- 或加权融合

------

## 2. 重排

混合召回出来几十条后，不直接送给 LLM，而是先 rerank。

常见做法：

- cross-encoder reranker
- 对 query + chunk 做相关性打分

这样 Top-1 准确率提升 13% 就很合理，因为它本来就是 reranker 最擅长拉的指标。

------

## 3. 二次过滤机制

这段描述很关键，因为它不是普通 RAG 都会做的。

常见过滤规则会包括：

- 去掉 OCR 置信度太低的 chunk
- 去掉重复 chunk
- 去掉和 query 关键词完全无交集的低价值 chunk
- 要求多个 chunk 证据一致
- 对医学领域保留证型/方剂/病机实体命中更高的 chunk

也就是说，这一层不是“再检索一次”，而是：

**在进入生成前，再把低质量证据清掉。**

这就是为什么你能把：

- OCR 干扰压下去
- Recall@30 拉高
- Top-1 也拉上去

------

## 4. P95 120ms 是什么含义

这通常只是在说：

**Faiss-HNSW 检索阶段的延迟**

一般不包括：

- LLM 推理
- reranker 全流程
- 最终回答生成

所以这类指标更多是：

- 检索工程实现得还不错
- 不是整个问答端到端只要 120ms

------

# 四、第三段：动态检索决策 Agent 是怎么落地的

你写的是：

> 基于 CoT+ReAct 构建动态检索决策 Agent，将静态检索升级为“逐轮判断”流程，并基于证据一致性与置信度控制检索。

实际落地时，这通常**不会真做成很重的自由 Agent**，而更像一个**状态机式 Agent**。

## 真实可落地的流程通常是：

### Step 1：判断问题类型

例如：

- 事实问答
- 汇总问答
- 多跳问答
- 是否需要继续检索

### Step 2：首轮检索

- dense + sparse
- rerank
- evidence judge

### Step 3：证据判断

判断当前证据是否足够：

- top1 分数是否够高
- top3 是否一致
- 是否命中关键实体
- 是否相互矛盾

### Step 4：不够则继续动作

动作通常只有几种：

- rewrite query
- 扩大 top-k
- 改检索粒度
- 切到章节级块
- 重新召回一次

### Step 5：足够则回答，否则拒答

这就是“逐轮判断”。

------

## CoT + ReAct 在真实项目里怎么用

通常不会真的把全部链路都变成长链推理，而是：

- **CoT** 用在 evidence judge 和 answer synthesis
- **ReAct** 用在动作选择上

也就是说，它更像：

**LLM 决定“现在该检索、改写、回答还是拒答”**

而不是每一步都做超长思维链。

------

## “基于证据一致性与置信度控制检索”怎么实现

常见实现会有一个综合分数：

```
confidence = 
  rerank_top1_score
+ topk_consistency_score
+ entity_match_score
- ocr_noise_penalty
- contradiction_penalty
```

然后：

- 高于阈值：回答
- 中间区域：再检一次
- 太低：拒答

这就是 Agent 控制逻辑的核心。

# 五、第四段：分层记忆是怎么实现的

你写的是：

> 构建分层记忆：短期记忆用滑动窗口+摘要压缩管理上下文，importance 由医疗 NER 加权并随轮次衰减；长期记忆按 BGE 相似度 + importance 阈值筛选入库，结合 NLI 冲突检测控制写入。命中率升 10%，错误率降 13%。

这部分是整个描述里**最像高级扩展项**的一段，也是实际工程里最容易做重的一段。

------

## 1. 短期记忆

一般不是“无限保留聊天记录”，而是分两层：

### 滑动窗口

保留最近几轮原始对话。

### 摘要压缩

更早的对话不保留原文，而是压成 summary。

这样做的目的：

- 控制上下文长度
- 保留连续问答的重点

------

## 2. importance 计算

你写的“医疗 NER 加权并随轮次衰减”，实际就是：

### importance = 领域实体权重 + 交互重要度 - 时间衰减

例如：

- 提到证型、病机、方剂、症状 -> 加分
- 用户明确纠错或确认 -> 加分
- 距离当前轮次越远 -> 衰减

伪公式大概会像：

```
importance = 
  a * medical_ner_score
+ b * query_relevance
+ c * user_feedback
- d * turn_decay
```

------

## 3. 长期记忆入库

不是所有内容都入长期记忆，而是要过筛：

### 筛选标准

- 和历史知识向量相似度高或能补充知识
- importance 超过阈值
- 不是明显重复
- 不和已有知识冲突

------

## 4. NLI 冲突检测

这部分是高级点。

实现方式通常是：

- 新记忆 vs 已有记忆
- 做 entailment / contradiction / neutral 判断
- 如果冲突：
  - 拒绝写入
  - 或标记旧知识过期
  - 或进入人工检查

这就能减少“记错了还写进长期记忆”的问题。

------

## 5. 这部分的真实难点

它最难的不是写代码，而是：

- importance 怎么定义
- 记忆什么时候读
- 记忆什么时候写
- 冲突怎么处理
- 命中率提升是不是记忆带来的，而不是检索本身提升带来的

所以如果这真是一个实际项目，记忆部分很可能是：

- 在主链路稳定后做的
- 甚至只是对多轮场景子集做的实验

# 六、第五段：RAGAS 评测闭环是怎么实现的

你写的是：

> 基于 RAGAS 构建自动化评估闭环，在 700 条测试集上系统忠实度由 71% 提升至 85%，上下文精确度与召回率均超过 90%，并支持证据引用与结果可追溯分析。

这部分工程上通常分成 3 层。

## 1. 构造测试集

700 条不是随便问出来的，一般会标：

- question
- reference answer
- gold evidence
- question type
- 是否应拒答
- 来源文档页码

这一步很费时间，但没有它，后面评测都不稳。

------

## 2. 自动评测

RAGAS 常测这些：

- Faithfulness
- Context Precision
- Context Recall
- Answer Relevance

你这里的：

- 忠实度 71% -> 85%
- 上下文精确度 / 召回率 > 90%

基本就是这套体系跑出来的。

------

## 3. 结果可追溯分析

真正工程化的做法，不是只看一个总分，而是把每次回答都留日志：

- query
- retrieved chunks
- rerank score
- final evidence
- answer
- citations
- whether rewritten
- memory hit or not

这样你才能做错误归因：

- 是解析错了
- 是 OCR 污染
- 是检索漏召
- 是 Agent 判断错了
- 是 memory 写坏了
- 是生成胡编了

这就是“闭环”。

# 八、这段项目介绍里，哪些部分最像“真的做过”

从工程可落地角度看，最真实、最常见的是这几段：

### 很像真实做过的

- 多格式解析 + 语义切分 + token 回退
- hybrid retrieval + rerank + 二次过滤
- RAGAS 评测 + 700 条测试集
- Faiss-HNSW P95 latency

这些都非常工程化。

### 可能做过，但实现会比话术收缩一些的

- CoT + ReAct 动态检索 Agent
- 证据一致性与置信度控制
- 分层记忆
- NLI 冲突检测

这些大概率不是“一个超完整通用系统”，而是：

- 针对特定场景子集做的
- 以规则 + LLM 混合实现的
- 在实验中验证过，但未必是全量上线主链路

------

# 九、一句话总结这段项目在真实工程里是怎么落地的

这段项目如果真实落地，最可能的实现方式是：

**先用结构化解析 + 混合检索 + rerank + 证据过滤把主链路打稳，再在此基础上叠加一个轻量状态机式 Agent 控制检索，并在多轮场景中加入短期摘要记忆和长期向量记忆，最后通过 RAGAS + 检索指标 + 错误归因日志形成评测闭环。**
